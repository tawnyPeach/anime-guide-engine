# HTML rendering fix, expanded filler database, and two new seed scripts

This change fixes a rendering problem in the content generator (switching from markdown notation to proper HTML with Tailwind classes), massively expands the hardcoded filler database from 10 to 107 entries, and adds two new seed scripts: one that scrapes animefillerlist.com for filler data and one that batch-fetches anime from the AniList API with resume support.

Watch for: The content generator now injects `anime.description` directly into HTML template literals without sanitization (confirmed). The filler scraper includes a User-Agent string that identifies as a bot, which may get blocked or violate ToS (confirmed). The resume heuristic in `seed-anime.ts` can re-process pages if anime are deleted from the DB (likely).

## High-level view

The content-generator rewrite replaces markdown template literals with HTML template literals containing Tailwind classes. This eliminates the `.replace(/\n/g, "<br/>")` hack that was spread across six page components. The output now renders structured headings, lists, and paragraphs directly. The risk is that `anime.description` and `anime.title` (sourced from AniList) are interpolated into raw HTML without escaping.

The filler database expansion is a static data dump: 97 new entries added to the `FILLER_DATABASE` array in `filler-data.ts`, all with empty episode arrays. The runtime function `fetchFillerDataFromGitHub()` that previously fetched from a GitHub JSON file now simply returns the local array. Actual filler episode numbers come from a separate scraping step.

`seed-filler.ts` introduces a cheerio-based scraper that hits animefillerlist.com for 190+ slug variations, matches them against existing anime in the database, and upserts `FillerMapping` records plus individual `Episode` flag updates. It rate-limits to 1 request/second.

`seed-anime.ts` is a resumable batch fetcher that pages through AniList's API (sorted by popularity), upserting anime records, creating placeholder episodes, and linking relations. It auto-detects where to resume by counting existing rows, which works when the DB only grows but breaks if records are removed between runs.

<summary>Issues (5)</summary>

1. **Unsanitized HTML interpolation** — `anime.description` and `anime.title` from AniList are injected into HTML template strings without escaping in `content-generator.ts`. Sanitize or escape these values before interpolation to prevent stored XSS if AniList data contains HTML/script payloads.
2. **Resume heuristic drift** — `seed-anime.ts` calculates start page as `floor(count / 50) + 1`. If anime are deleted, this under-counts and re-fetches pages already processed, which is harmless but wasteful. More critically, if the AniList sort order changes between runs, the script may create duplicate slugs or miss entries. Consider tracking the last-fetched page explicitly.
3. **Scraper fragility** — `seed-filler.ts` tries three different CSS selector strategies for animefillerlist.com, suggesting the page structure is inconsistent or unknown. If none match, the script logs a warning and moves on, meaning many entries may silently fail. No integration test validates the scraping logic.
4. **Production dependency on cheerio** — `cheerio` is added as a production dependency in `package.json` but is only used in the seed script (a dev-time tool). It should be a `devDependency` to keep the production bundle smaller.
5. **Empty filler arrays provide no runtime value** — 97 of the 107 entries in `FILLER_DATABASE` have empty `fillerEpisodes`, `mixedCanonFillerEpisodes`, and `canonEpisodes` arrays. They serve as slug-matching anchors for the scraper but add ~700 lines of static data that `fetchFillerDataFromGitHub()` returns to callers with no useful information until the scrape has run.

## Unsanitized description injection in content-generator

The `generateEpisodeGuideContent` function interpolates `anime.description` directly into an HTML string:

```typescript
const aboutText = anime.description || `${title} is a popular anime series...`;
// later:
return `...
<p class="text-gray-300 leading-relaxed mb-4">${aboutText}</p>
...`;
```

`generateAnimePageContent` does the same with `descText`. AniList's API returns descriptions that already contain HTML (`<br>`, `<i>`, `<b>` tags), and while `normalizeAniListMedia` strips tags with `.replace(/<[^>]*>/g, "")`, that regex-based strip is not a sanitizer — it can be bypassed with malformed tags like `<img src=x onerror=alert(1)>` split across encoded characters. Since this HTML lands in `dangerouslySetInnerHTML`, any bypass results in stored XSS on every page that renders the anime description.

The `anime.title` fields are similarly unescaped, though the risk is lower since titles are shorter and more constrained by AniList's validation.

## Resume heuristic in seed-anime assumes monotonic growth

```typescript
async function determineStartPage(): Promise<number> {
  const existingCount = await prisma.anime.count();
  const calculatedPage = Math.floor(existingCount / 50) + 1;
  ...
}
```

This works under two assumptions: (1) the database never shrinks, and (2) AniList's popularity sort order is stable between runs. If the sort order shifts (new anime gains popularity and displaces older entries), the page boundaries no longer align with what was previously fetched. The upsert on `anilistId` prevents duplicates, but anime that shifted to earlier pages will never be fetched if those pages are already "done." This is acceptable for a seed script but worth documenting.

## Scraper selector brittleness and silent failure

`seed-filler.ts` attempts three CSS selector patterns sequentially:

1. `.episodes .ep` with class-based filler detection
2. `tr.filler span.Label` / `td.filler` 
3. `.EpisodeList .list-group-item` / `.EpisodeList tr`

If all three fail to match any elements, the result is `totalEpisodes: 0` and the entry is logged as `[WARN] No episode data found` and skipped. For a list of 190+ slugs, a silent failure rate is hard to gauge without running the script. The slug list includes duplicates (`hell-teacher-nube` appears twice) and speculative entries like `boruto-two-blue-vortex` that likely don't exist on animefillerlist.com yet.

The `findAnimeBySlug` function in the same file does partial/title-keyword matching, which could match the wrong anime in the database (e.g., `shield-hero` matching a different "Shield" title). The `contains` query without word boundaries is imprecise for short slugs.

## cheerio as a production dependency

```json
"dependencies": {
  "@prisma/client": "^5.22.0",
  "cheerio": "^1.2.0",
  ...
}
```

cheerio (and its dependency tree: htmlparser2, dom-serializer, parse5, undici, etc.) adds ~300 lines to `package-lock.json` and ships in the production `node_modules`. Since it's only imported by `scripts/seed-filler.ts` (a CLI tool run locally or in CI), it belongs in `devDependencies`.

## Removed GitHub fetch in favor of static return

The previous `fetchFillerDataFromGitHub()` made a network request to a GitHub raw file and fell back to the local database on failure. The new implementation unconditionally returns `FILLER_DATABASE`. This simplifies the runtime path (no network call, no error handling) but means the 97 entries with empty arrays are served as-is to any caller until the scrape script populates `FillerMapping` records separately. The function name is now misleading — it doesn't fetch from GitHub.
