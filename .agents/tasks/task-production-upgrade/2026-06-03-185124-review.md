# Production upgrade: 1000-anime seed, cron sync, new page types, and client-side features

This change expands the anime guide engine from a 200-anime static site into a broader content platform. The seed script now targets 1000 anime with episode titles from Jikan, a Vercel Cron route keeps the top 250 fresh weekly, and five new page types (studio, comparison, season, top lists, blog) dramatically increase the SEO surface. On the client side, dark/light theming, localStorage-based bookmarks and episode tracking, share buttons, and placeholder integrations (Giscus comments, newsletter) round out the user-facing feature set. An in-memory cache with stale-while-revalidate wraps the API routes.

**Watch for:** The cron sync endpoint's auth check rejects all requests when `CRON_SECRET` is unset, meaning a misconfigured deployment silently fails every weekly sync (confirmed). The in-memory cache lives per-serverless-instance and will be effectively empty on every cold start in a Vercel deployment, making the stale-while-revalidate semantics mostly theoretical in production (confirmed). The Newsletter component shows a "Thanks for subscribing!" message without ever sending data anywhere (confirmed). The OG image route accepts arbitrary user-supplied strings rendered into an SVG context without sanitization (likely).

## High-level view

The cron sync route is the most operationally significant addition. It fetches from AniList, GitHub (filler data), and Jikan within a single Vercel function invocation, using a 4.5-minute timeout budget that it checks between phases. The auth model is a `CRON_SECRET` bearer token compared against `process.env`. If the secret is empty or unset, the route returns 401 to all callers including Vercel's own cron scheduler, which means a misconfigured deployment silently fails every sync.

The in-memory `cached()` utility wraps Prisma queries in both API routes. Because Vercel serverless functions spin up fresh instances per invocation (or at least per cold start), this cache will rarely hold stale data to serve. The background-refresh promise fires and resolves, but on the next request a different instance likely handles it. The cache Map also grows without eviction — every unique search query adds an entry that lives until the instance is recycled.

The recommendations engine replaces the old "shared genre count" approach with a weighted multi-signal scorer (genre overlap, format match, studio match, year proximity, normalized popularity). It fetches 200 candidates ordered by popularity from Prisma. As the dataset grows past 1000, the popularity-ordered `take: 200` will silently exclude relevant-but-niche matches from scoring, degrading recommendations for less popular source anime.

New page types (season, studio, top lists, comparison) follow the same ISR pattern with graceful empty-state fallback. The comparison page splits the dynamic segment on `-vs-`, which breaks for anime whose slugs contain that literal substring (e.g., slug `angels-vs-demons` would cause a three-way split and 404).

Client-side features (theme toggle, bookmarks, episode tracker) use `useSyncExternalStore` with module-level listener arrays and server-safe default snapshots. The ThemeProvider mutates `document.documentElement.classList` during render rather than in an effect, violating React's render purity contract.

## Cron sync: timeout budget and auth gap

The auth check compares `authorization: Bearer <secret>` against `process.env.CRON_SECRET`:

```typescript
if (!cronSecret || authHeader !== `Bearer ${cronSecret}`) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

When `CRON_SECRET` is empty string (the `.env.example` default), `!cronSecret` is `true`, so the route always returns 401. Vercel Cron sends the `Authorization` header automatically only if the secret is configured in the project settings. A developer who deploys without setting the secret will see the sync silently fail with 401 every Sunday at 3 AM, with no logging or alerting.

The Jikan pagination loop has a retry-on-429 path that `continue`s without incrementing the page counter. However, there's no cap on consecutive 429 retries, so a sustained Jikan rate-limit could keep the loop spinning until the timeout budget is exhausted on a single anime.

## In-memory cache on serverless

```typescript
const cache = new Map<string, CacheEntry<unknown>>();
```

This module-level `Map` persists across requests only if the same instance handles them. The stale-while-revalidate pattern fires a background `fn()` call, but if the next request hits a different instance, that instance has an empty cache and awaits fresh data anyway. The pattern would be effective on a long-lived server (Node custom server, Docker) but provides minimal benefit on Vercel's default serverless deployment model.

The cache has no eviction. Every unique search query (`search:${q}`) adds an entry that lives until the instance is recycled. Under bot traffic or broad organic search, memory could grow meaningfully within a warm instance's lifetime.

## OG image generation and input handling

The `/api/og` route runs on the edge runtime and passes `title`, `subtitle`, and `type` directly from query params into JSX consumed by `ImageResponse`. The `ImageResponse` API uses Satori under the hood, which renders JSX to SVG. Satori generally handles text content safely, but characters like `<`, `>`, `&` in titles haven't been verified to render correctly. Anime titles regularly contain ampersands and special characters, and the URL is publicly accessible with arbitrary query params.

## ThemeProvider render-time DOM mutation

```typescript
if (typeof window !== 'undefined') {
  const current = getThemeSnapshot();
  document.documentElement.classList.toggle('light', current === 'light');
  document.documentElement.classList.toggle('dark', current === 'dark');
}
```

This runs on every render of the `ThemeProvider` component, which wraps the entire app. In React 18's concurrent mode, renders can be interrupted and restarted, meaning this side-effect could execute multiple times or at unexpected moments. The correct placement is inside a `useLayoutEffect` to apply the theme class synchronously after commit without violating render purity.

## Recommendations engine: 200-candidate cap

```typescript
const candidates = await prisma.anime.findMany({
  where: {
    id: { not: animeId },
    OR: sourceGenres.map((genre) => ({ genres: { contains: genre } })),
  },
  orderBy: { popularity: "desc" },
  take: 200,
});
```

The `take: 200` combined with `orderBy: { popularity: "desc" }` means recommendations for niche anime will always be dominated by popular titles that happen to share a genre, drowning out more relevant but less popular matches. The weighted scoring (studio match, format match, year proximity) only helps if the relevant candidates make it past the popularity filter. At 1000 anime this is unlikely to be visible; at 5000+ it will matter.

## Newsletter: UI without backend

The Newsletter component's submit handler sets local state to show a success message without making any network request. The inline comment says "TODO: Connect to Buttondown API", but the UI gives no indication to the user that this is a placeholder. A user who types their email and clicks "Subscribe" will believe they're subscribed.

## Comparison page slug parsing

```typescript
const parts = slugs.split("-vs-");
if (parts.length !== 2) notFound();
```

Current slug generation strips non-alphanumeric characters except hyphens, so a title like "Angels vs Demons" produces slug `angels-vs-demons`. A comparison URL `angels-vs-demons-vs-another-anime` would split into three parts and 404. The probability is low with current data but non-zero.
