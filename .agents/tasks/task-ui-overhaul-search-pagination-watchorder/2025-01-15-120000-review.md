# UI overhaul, search, pagination, and franchise watch order builder

This change transforms the Anime Guide Engine from a basic dark-gray layout into a neon-accented anime-themed site while adding three functional features: header search with debounced Prisma queries, cursor-free "Load More" pagination on all listing pages, and a franchise watch order system that BFS-traverses the relation graph then topologically sorts SEQUEL/PREQUEL chains. The visual overhaul touches globals.css (custom properties, glow utilities, keyframe animations, glass and gradient-mesh classes), every page template, and the AnimeCard/Header/Footer components.

**Watch for:** The search API has no rate limiting or input sanitization beyond a minimum-length check â€” three `contains` queries fire against SQLite on every keystroke after debounce (confirmed). The watch order builder issues N+1-style sequential batched queries during BFS, which could hit latency walls on large franchises at build time (likely). The pagination API accepts arbitrary `limit` values from the client with no upper bound (confirmed).

## High-level view

The visual layer replaces hard-coded `bg-gray-800`/`bg-gray-700` classes with a design-token palette (`anime-dark`, `anime-card`, `anime-border`, `neon-*`) defined via TailwindCSS 4's `@theme inline`. Custom utility classes like `.glass`, `.gradient-text`, and `.glow-card-hover` give the templates a consistent vocabulary. The palette isn't user-switchable â€” CSS-only, no runtime theme context.

Search is a client-only autocomplete: `SearchBar` debounces at 300ms, calls `/api/search?q=...`, and renders a positioned dropdown. The API does a three-column `contains` match against `title`, `titleEnglish`, `titleJapanese` and returns the top 10 by popularity. No server-side request validation beyond the 2-character minimum â€” no upper length bound on the query string.

Pagination avoids changing the SSG strategy. Pages render the first 20 items server-side, then a `LoadMore` client component fetches subsequent pages from `/api/anime`. The count shown to users (`totalPopular`) uses an unfiltered `prisma.anime.count()` on the homepage â€” identical to `totalAnime`, making one variable redundant and the "showing X of Y" label fragile if filters are added later.

The watch order builder (`src/lib/watch-order-builder.ts`) discovers the entire franchise via BFS over `animeRelation` edges, then builds a directed "comes before" graph from SEQUEL/PREQUEL relations and runs Kahn's algorithm to produce a linearized order. Side stories, OVAs, and spin-offs land in a supplementary bucket. Cycle handling appends stranded nodes sorted by year â€” functional but silent about the cycle.

The header's mobile breakpoint renders the search bar but hides all navigation links with no hamburger menu, despite this being listed in FEAT-001's acceptance criteria.

## Pagination API surface and client contract

The `/api/anime` route accepts `page`, `limit`, `genre`, `year`, and `sort` from the query string. The `limit` parameter is parsed directly with no upper bound:

```ts
const limit = parseInt(searchParams.get("limit") || "20", 10);
```

A bot can pass `?limit=99999` and receive the full table in one response. For SQLite this won't crash today, but it's a free DoS vector if the database grows or the backend migrates to a hosted DB with connection limits.

The `genre` filter uses Prisma `contains` on a JSON-serialized string column. This is a substring match â€” searching for "Action" will also match "Live Action" or any genre whose name contains that substring. Latent ambiguity that depends on the genre vocabulary staying clean.

## Search: race condition and backend exposure

The `SearchBar` debounces with `setTimeout` but doesn't abort prior `fetch` calls. If the user types "nar" â†’ waits 300ms â†’ types "naruto", both requests fire and the earlier response can briefly flash stale results before being overwritten. Benign but not eliminated â€” an `AbortController` would close the gap.

On the backend, `/api/search` has no upper bound on query length:

```ts
const q = searchParams.get("q")?.trim() ?? "";
if (q.length < 2) return Response.json({ results: [] });
```

URL length limits in HTTP servers provide an incidental cap, but the code doesn't enforce one explicitly.

## Watch order graph traversal and cycle handling

The BFS processes the queue in levels, issuing a `prisma.animeRelation.findMany` per level with an `OR` condition on `fromAnimeId`/`toAnimeId`. For a franchise spanning 5+ BFS levels (long-running shounen are common), this serializes 5+ sequential DB calls at build time. A two-phase approach â€” discover all node IDs first, then fetch all edges for those IDs in one query â€” would reduce to 2 calls regardless of depth.

The topological sort uses Kahn's algorithm with year-based tiebreakers. Cycle-stranded nodes are appended without any signal:

```ts
for (const nodeId of mainNodeIds) {
  if (!topoVisited.has(nodeId)) {
    mainOrder.push(nodeId);
  }
}
```

This produces a "best effort" order that's silently degraded. Adding a `hasCycle: boolean` field to `WatchOrderResult` would let the page display a disclaimer.

## `dangerouslySetInnerHTML` and content injection

Both `watch-order/page.tsx` and `genre/[genre]/page.tsx` render content through `dangerouslySetInnerHTML` after a naive newline-to-`<br>` transform. If `content-generator.ts` interpolates anime titles from the database without HTML-escaping them, and an AniList title ever contains angle brackets, the markup will be interpreted. Worth verifying the generator's escaping behavior â€” a single `escapeHtml()` call on interpolated values would close this.

## Mobile navigation gap

The header uses `hidden md:flex` for the genre nav links and provides no alternative for viewports below `md`. The mobile breakpoint renders only the search bar. FEAT-001's step 3 specified "add a mobile hamburger menu toggle (client component)" and the acceptance criteria included it, but the implementation doesn't include any mobile nav.
