# Five-feature expansion: search, top lists, genre pages, "After" page, and AnimeCard enhancements

This diff adds five user-facing features to the AniYume anime guide site: a dedicated search page with advanced filters, three new "top" list types with a filter/pagination client component, enhanced genre pages with sub-genre filtering and curated sections, a new `/after/[slug]` page showing sequels and recommendations, and AnimeCard cosmetic improvements (filler badge, pulsing AIRING dot, episode count format). Three new API routes (`/api/search/advanced`, `/api/top`, `/api/genre`) back the client-side filtering components, and the sitemap gains entries for the new top types and `/after/` pages.

**Watch for:** The genre and search API routes accept arbitrary user-controlled strings passed directly into Prisma `contains` filters without length or character validation (confirmed). The `/api/top` route lacks page-number bounds checking, allowing skip values that force full-table scans (confirmed). The in-memory cache (`Map`) grows without per-route isolation and the 1000-entry eviction threshold is shared across all API consumers, making cache poisoning trivial via enumeration of unique query param combinations (likely).

## High-level view

The search page is a fully client-rendered React component with debounced fetch, localStorage-backed recent searches, and a filter panel. It talks to a new `/api/search/advanced` route that builds Prisma WHERE clauses from seven optional query params. The approach works but offers no server-side rendering or URL-based state, so search results are invisible to crawlers and not shareable via URL.

The top-lists expansion adds three new list types (currently-airing, most-filler, least-filler) as static pages and layers a `TopFilterBar` client component that calls `/api/top` for client-side filtering and pagination. The server-rendered page loads the first 100 items; the filter bar replaces them entirely once any filter is activated, creating two uncoordinated data sources on the same page.

The genre page enhancements introduce parallel Prisma queries for a "Currently Airing" strip and a "Hidden Gems" section, both server-rendered, plus a `GenrePageClient` component for dynamic sub-genre/year/format filtering via `/api/genre`. The genre-specific prose content is expanded to four distinct multi-paragraph sections per genre (intro, definition, why-fans-love-it, tips), all generated from static lookup maps in `content-generator.ts`.

The `/after/[slug]` page is a server component that queries anime relations (SEQUEL, SIDE_STORY, SPIN_OFF) and delegates recommendations to the existing `getRecommendations` function. It includes structured data (JSON-LD `ItemList`), breadcrumbs, and sitemap integration. The sitemap query for `/after/` pages uses a broad `OR` filter on `relationsFrom`/`relationsTo` which may generate a large result set without pagination.

The AnimeCard changes are purely additive: two optional props (`fillerCount`, `fillerPercent`) drive a color-coded filler badge overlay and an augmented episode-count label. The AIRING badge gains a pulsing dot. Footer years and sidebar search link are trivial one-line updates.

<summary>Issues (7)</summary>

1. **Unbounded genre/search string input** — The `/api/genre` and `/api/search/advanced` routes pass user-supplied strings into Prisma `contains` without length limits (genre route) or character sanitization. Apply `.slice(0, 100)` and reject empty single-character strings to prevent expensive LIKE scans.
2. **Unbounded page parameter in /api/top** — `page` is parsed with `parseInt` but never clamped to a maximum. A request with `page=999999` produces a massive `skip` value, forcing the database to scan and discard rows. Add `Math.max(1, Math.min(page, 500))` or equivalent.
3. **Cache poisoning via param enumeration** — All three API routes share a single 1000-entry in-memory `Map`. An attacker can evict legitimate cache entries by requesting unique param combinations (random year values, page numbers). Consider per-route cache namespaces with independent size limits, or rate-limit cache insertions.
4. **Search page not crawlable** — `SearchPageClient` renders results entirely client-side with no URL state. Search results are invisible to search engines and not shareable. Consider syncing filters to query params via `useSearchParams` and rendering a server-side fallback for the initial page load.
5. **TopFilterBar replaces server-rendered content** — Once a user interacts with any filter, the `TopFilterBar` renders its own grid below the existing SSR grid, creating a confusing dual-grid layout. The SSR grid of 100 items remains visible above. Consider hiding the SSR grid when `hasFiltered` is true, or integrating the filter into the SSR flow.
6. **Sitemap unbounded /after/ query** — The sitemap fetches all anime with at least one relation (`relationsFrom: { some: {} }`) without a `take` limit. For a database with thousands of related anime, this produces a very large sitemap segment and an expensive query on every sitemap generation.
7. **Missing `revalidate` on new API routes** — The API routes rely on the in-memory `cached()` helper with 60s TTL, but Next.js Route Handlers default to dynamic (no CDN cache). Consider adding `export const revalidate` or appropriate `Cache-Control` headers so the CDN layer reduces origin load.

## API route input validation gaps

The `/api/genre` route accepts `genre` as a raw query param and passes it directly to `where.genres = { contains: genre }`. Unlike the search route (which slices the query to 200 chars), the genre route applies no length constraint. An adversary can pass a multi-KB string into the `LIKE` clause:

```ts
// /api/genre/route.ts
const genre = searchParams.get("genre") || "";
// ...
where.genres = { contains: genre }; // no length check
```

The search route fares slightly better with `q?.trim().slice(0, 200)` on the text query, but `genre`, `format`, and `status` params pass through unchecked. Since these end up in `contains` (case-insensitive LIKE) Prisma clauses, arbitrarily long strings force the database to scan with expensive pattern matching.

The `page` parameter in `/api/top` uses `parseInt` without an upper bound:

```ts
const page = parseInt(searchParams.get("page") || "1", 10);
const skip = (page - 1) * limit;
```

A request to `/api/top?page=999999&limit=50` produces `skip = 49999950`, which PostgreSQL must evaluate by scanning and discarding that many rows before returning the final page. The other two routes share the same pattern (`Math.max(1, ...)` without a max).

## In-memory cache under adversarial load

The `cached()` utility stores entries in a module-level `Map` with a hard cap of 1000 entries. All three new API routes (plus any pre-existing consumers) share this namespace. Cache keys include every filter parameter:

```ts
const cacheKey = `top:${type}:${page}:${limit}:${genre || ""}:${yearParam || ""}:${format || ""}:${sort || ""}`;
```

An attacker can enumerate unique keys (varying `page`, `year`, `genre` in combinations) to fill the cache, triggering the eviction path that removes the 200 oldest entries. Legitimate hot entries for popular queries get evicted, forcing repeated database hits. The 60-second TTL means even under normal traffic, a burst of unique searches purges the cache for everyone.

## TopFilterBar dual-grid problem

The server-rendered top page fetches 100 anime and renders them in a grid. Below that grid, `TopFilterBar` sits dormant until a filter is selected. Once `hasFiltered` becomes true, the component renders its own independent grid from API data. Both grids are visible simultaneously — the SSR grid above and the client-filtered grid below. There's no mechanism to hide or replace the SSR content:

```tsx
{/* In top/[type]/page.tsx */}
{/* SSR grid with 100 items always visible */}
<div className="grid ...">
  {allAnime.map((anime) => <AnimeCard ... />)}
</div>

{/* Client filter bar renders below */}
<TopFilterBar type={type} />
```

A user applying a genre filter sees two potentially contradictory result sets on the same page.

## /after/[slug] structured data and recommendations

The page emits JSON-LD `ItemList` structured data combining sequels, side stories, and recommendations into a single list. The list positions are sequential regardless of category, which is semantically correct for Google but somewhat misleading — position 1 might be a sequel while position 5 is a recommendation from a different source. The `getRecommendations` call is awaited after the relation queries rather than parallelized:

```ts
const sequelRelations = await prisma.animeRelation.findMany({ ... });
const sideStoryRelations = await prisma.animeRelation.findMany({ ... });
const recommendations = await getRecommendations(anime.id, 10);
```

These three queries are sequential when they could be parallel with `Promise.all`, adding latency proportional to the sum of all three query times rather than the max.

## Genre page content expansion

The `content-generator.ts` additions add approximately 130 lines of static prose per genre across four functions (`getGenreIntro`, `getGenreDefines`, `getGenreWhyFansLoveIt`, `getGenreTips`). The HTML returned includes hardcoded Tailwind classes (`text-gray-300`, `text-purple-400`) which couple content generation to a specific color theme. If the site's theme changes (as it already has — the codebase uses `text-foreground`/`text-muted-foreground` semantic tokens elsewhere), these embedded classes will drift out of sync. The genre descriptions inject `animeCount` via template literal into raw HTML strings, which is safe for integers but the pattern of building HTML via string concatenation is fragile for future maintenance if non-numeric data gets interpolated.

## Sitemap scale

The sitemap now executes two unbounded queries: the existing `prisma.anime.findMany` (all anime) and a new `prisma.anime.findMany` with `OR: [{ relationsFrom: { some: {} } }, { relationsTo: { some: {} } }]`. Neither has a `take` limit. For a database with 5000+ anime (which the project's seed script targets), the sitemap generation produces thousands of URLs per category. The XML sitemap spec caps at 50,000 URLs per file; the total here may approach that limit given anime + filler + episodes + watch-order + anime-like + genre + year + studio + season + top + compare + after pages. No sitemap index splitting is implemented.
