# Filler timeline minimap and AniList-powered calendar

Two user-facing features land in this change: a color-coded episode minimap rendered inline on filler-list pages, and a full weekly airing calendar sourced from AniList's public GraphQL API. The minimap is self-contained (no network calls, pure data visualization). The calendar introduces an external dependency on AniList's rate-limited API that feeds both a dedicated `/calendar` page and a "Currently Airing" strip on the homepage. Both paths are behind try/catch with graceful degradation.

**Watch for:** Unbounded retry recursion in the AniList fetch helper (confirmed). UTC-vs-local timezone mismatch between server-side day grouping and client-side "today" detection (confirmed). The homepage becomes coupled to AniList availability during ISR builds, potentially adding seconds to revalidation latency (likely).

## High-level view

The calendar data layer (`src/lib/calendar.ts`) paginates AniList's airing schedule endpoint, groups entries by UTC day-of-week, and exposes two functions: one for the full week, one for today only. A rate-limit retry is attempted recursively without a depth limit — the primary reliability gap in this change.

The calendar page is a server component with ISR at 1 hour. The client-side `CalendarGrid` handles day-tab switching and a 60-second interval tick for countdown freshness. An API route at `/api/calendar` wraps the same fetch with an in-memory cache, but the calendar page doesn't use it — it calls `fetchWeeklySchedule` directly during SSR, making the route dead code.

The filler timeline takes pre-computed episode data as props and renders a horizontal strip of colored blocks with arc grouping, tooltips, and a clipboard share button. Block width scales by episode count to handle series ranging from 12 to 1000+ episodes.

The homepage gains a "Currently Airing" section that fetches today's AniList schedule at build/revalidation time. This adds a potentially slow network call to every ISR cycle of the most critical page, with no fetch timeout.

<summary>Issues (5)</summary>

1. **Unbounded retry recursion** — `fetchAiringPage` calls itself recursively on HTTP 429 with no depth counter. If AniList keeps returning 429, the server recurses until stack overflow or timeout. Add a max-retry parameter (e.g., 3 attempts).
2. **UTC/local timezone mismatch** — The server groups schedule entries into days using UTC day-of-week, but `CalendarGrid` highlights "today" using the browser's local timezone. A user in UTC+9 on a Tuesday evening will see their local Wednesday content filed under "Tuesday". Either group on the client or pass timezone context.
3. **Dead API route** — `/api/calendar` exists with its own caching layer but nothing in the app consumes it. The calendar page calls `fetchWeeklySchedule` directly. Either remove the route or have the client fetch from it for SPA-style navigation.
4. **Homepage ISR latency coupling** — `fetchTodaySchedule` runs during ISR revalidation of the homepage. If AniList is slow or rate-limits, homepage regeneration blocks for up to 60 seconds (the default Retry-After). Consider adding a timeout or moving this to a client-side fetch.
5. **toLocaleTimeString SSR hydration mismatch** — The homepage renders `toLocaleTimeString` on the server (Node locale) and the client re-renders with the browser locale. This produces a React hydration warning for time strings. The CalendarGrid avoids this by being `'use client'`, but the homepage cards render times during SSR.

## AniList fetch layer and retry behavior

`fetchAiringPage` in `src/lib/calendar.ts` handles 429 responses by sleeping for the `Retry-After` header value then calling itself again. There's no recursion depth guard:

```typescript
if (response.status === 429) {
  const retryAfter = parseInt(response.headers.get("Retry-After") || "60", 10);
  await new Promise((resolve) => setTimeout(resolve, retryAfter * 1000));
  return fetchAiringPage(page, airingAtGreater, airingAtLesser); // unbounded
}
```

AniList's rate limit is 90 requests/minute, and `Retry-After` is typically 60 seconds. In a sustained rate-limit scenario (e.g., multiple concurrent ISR rebuilds hitting the same origin), this recurses indefinitely. A `maxRetries` counter defaulting to 3 would bound the behavior without changing the happy path.

## Timezone discrepancy between server and client

The server builds week bounds using UTC (`setUTCHours`, `getUTCDay`) and groups airing entries into day buckets by UTC day. The `CalendarGrid` client component determines "today" via `new Date().getDay()` — the browser's local timezone. For users east of UTC, entries airing late in their evening get assigned to the next UTC day but display under the previous local day tab. This manifests as "today" being highlighted but showing yesterday's content, or the reverse.

The fix is either: (a) recompute day grouping on the client from raw `airingAt` timestamps (preferred, since the client knows the local timezone), or (b) pass all entries ungrouped and let `CalendarGrid` bucket them locally.

## Unused API route

`/api/calendar` wraps `fetchWeeklySchedule` in the in-memory `cached()` helper with a 1h TTL. However, the calendar page imports `fetchWeeklySchedule` directly for its SSR render:

```typescript
// src/app/calendar/page.tsx
const data = await fetchWeeklySchedule();
```

The route is never referenced anywhere in the codebase. If it's scaffolded for future client-side fetching, a comment would help; otherwise it should be removed or actually consumed by the client component for SPA-style day switching.

## Homepage coupling to AniList availability

The homepage server component calls `fetchTodaySchedule()` during every ISR revalidation cycle. This function paginates up to 5 pages against AniList, waiting for each sequentially. If AniList is degraded:

- 429 with 60s Retry-After → homepage regeneration blocks for 60+ seconds per retry
- Timeout or DNS failure → the fetch throws, caught by try/catch, section omitted

The 60-second blocking case is the concern. Next.js ISR won't serve a stale page to users during this window (it waits for the regeneration to complete before swapping). A `signal: AbortSignal.timeout(5000)` on the fetch calls would cap worst-case latency at 5 seconds, failing fast and rendering without the section.

## FillerTimeline tooltip positioning under scroll

The tooltip uses `getBoundingClientRect()` relative to the parent's parent container and positions absolutely. When the user scrolls the `overflow-x-auto` container horizontally, the tooltip's `left` calculation uses screen-relative coordinates rather than scroll-offset-relative ones. For long-running series (Naruto, One Piece, Detective Conan at 1000+ episodes), where the minimap overflows and scrolls, tooltips near the scroll boundary may clip or misposition.

Confidence: possible — would need to test with a 1000+ episode series to confirm.

## Hydration mismatch on homepage time strings

The "Currently Airing" section on the homepage renders time strings server-side:

```typescript
{new Date(entry.airingAt * 1000).toLocaleTimeString([], {
  hour: "2-digit",
  minute: "2-digit",
})}
```

This runs during SSR with Node.js's locale and produces locale-specific output. When the client hydrates with a different locale or timezone, React logs a hydration mismatch warning. The homepage section should either suppress the time display during SSR or extract it into a small client component (as `CalendarGrid` already does with `formatLocalTime`).
